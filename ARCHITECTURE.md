# 技术架构方案设计 (Technical Architecture)

## 1. 约束条件与选型背景
基于业务早期极小规模内测的特性，系统设计以“敏捷开发、极速响应、低维护成本”为核心原则，不过度设计高并发架构。

*   **业务规模：** 极低并发，注册用户 5 人以内，日活单人 1-2 次，日总对话轮次在 100 轮左右。
*   **网络环境：** 中国大陆客户端网络直连，要求低延迟穿透。
*   **交互底线：** 英日双语，10轮回合制，不允许中英/中日夹杂，必须“极速秒回”。

## 2. 核心大模型选型 (LLM Matrix)

基于 [Artificial Analysis 2026 最新速度评测]，以及中国大陆网络可达性，建立主备双链路保障机制：

### 2.1 实时对话层 (极速响应优先)
要求 TTFT (Time To First Token) 控制在 300ms 以内，生成速度 > 100 Tokens/s。
*   **主链路 (Primary)：DeepSeek V4 Flash** (基于大陆节点直连，网络延迟最低，且模型推理速度极快)。
*   **备链路 (Fallback)：Gemini 3.5 Flash** (当主链路 API 抖动时无缝切换，但需考虑服务器到 Gemini 节点的海外网络连通性方案)。

### 2.2 异步复盘层 (推理能力优先)
对延迟无感，重点考察长上下文理解和多维（语法、用词）纠错逻辑。
*   **主链路 (Primary)：DeepSeek V4 Pro** (强大的逻辑分析和中文结构化输出能力，适合生成 JSST/雅思 级复盘报告)。
*   **备链路 (Fallback)：Gemini 3.1 Pro**。

## 3. 语音链路选型 (豆包大模型新版流式语音)

### 3.1 产品选型

*   **选型：火山引擎「豆包语音大模型」新版流式 API**
*   **产品优势：**
    *   端到端统一模型，ASR + TTS 一体化，无需分别对接两套服务。
    *   WebSocket 全双工长连接（`wss://openspeech.bytedance.com/api/v3/sauc/bigmodel`），延迟更低。
    *   原生支持 VAD（静音检测）、情绪感知合成、多语种（英/日/中）。
    *   单一 API Key 认证，运维更简单。

### 3.2 交互模式

*   **协议：** WebSocket 全双工流式。
*   **上行（用户语音 → 文本）：** 小程序通过业务服务器中转，每 100ms 发送 PCM 音频帧至豆包语音 WebSocket。模型内置 VAD，自动判断用户说话结束，返回识别文本。
*   **下行（AI 文本 → 语音）：** LLM 输出文本经**句级断句**后，发送给豆包语音合成接口，返回的音频流实时下发给小程序播放。

### 3.3 句级断句策略（不变）

鉴于 DeepSeek V4 Flash 吐字极快，逐 Token 送入 TTS 会导致语调不连贯。后端实施**缓冲断句策略**：
1.  持续接收 LLM 的流式输出。
2.  利用正则匹配标点（`,`, `.`, `?`, `!`, `。`, `、`）。
3.  切分出完整短句后，立刻送入豆包语音合成。
4.  首段音频几毫秒内返回，通过 WebSocket 下发给小程序播放。
5.  利用播放第一句的”物理时间”，后台继续处理后续句子，实现听觉上的”零等待秒回”。

### 3.4 与旧版 ASR/TTS 的差异说明

| 维度 | 旧版（分离式） | 新版（豆包统一模型） |
|---|---|---|
| 服务数量 | ASR + TTS 各一个 WebSocket | 统一一个 WebSocket 端点 |
| 认证 | AppID + Access Token + Cluster | 单一 API Key |
| VAD | 服务端额外配置 | 模型内置 |
| 情绪合成 | 需选音色 + 额外参数 | 原生支持情绪感知 |
| 延迟 | ASR→文本→TTS 串行 | 模型内合并优化，理论更低 |

## 4. 后端架构极简设计 (Backend Micro-Architecture)

考虑到仅 5 人使用，摒弃复杂的微服务和重量级消息队列。

*   **运行环境：** 单机 Node.js / Go 进程 (搭配 PM2 守护或简单的 Docker 容器)。
*   **协议层：**
    *   `/api/v1/...` (RESTful)：处理登录、拉取场景、获取历史复盘。
    *   `/ws/conversation` (WebSocket)：承载所有的核心对练流数据（音频流上行 -> ASR 结果 -> LLM -> TTS 音频流下发）。
*   **数据库：**
    *   MVP 选用 **SQLite + Prisma ORM**。通过 Prisma 兼容层保留 PostgreSQL 迁移能力。当注册用户 > 50 / 日对话 > 5000 时切换 PostgreSQL。
    *   数据表：`users` / `placement_results` / `sessions` / `turns` / `reviews`（详见 `prisma/schema.prisma`）。
*   **异步任务 (复盘生成)：**
    *   无需 RabbitMQ/Kafka。10 轮对话完成后，后端直接在 Node 进程中 `setTimeout` 或使用 `Promise` 异步发起 DeepSeek V4 Pro 的请求，算完直接 Update 数据库字段即可。

## 5. 小程序端交互设计 (Frontend)

*   **权限申请：** 首次进入要求麦克风权限。
*   **录音组件：** 弃用传统的录完再传，必须使用 `RecorderManager.onFrameRecorded` 进行帧级别拦截并发送。
*   **音频播放：** 使用 `WebAudioContext` 接收后端的 ArrayBuffer，进行 PCM/MP3 数据的拼接和连续播放。
*   **网络防抖：** WebSocket 断线重连机制，加入 `ping/pong` 心跳。
